# Rust Bootstrap Agent (V0) 開發日誌

這份日誌記錄了開發過程中遇到的關鍵挑戰以及我們採取的技術對策。

## 📅 開發階段與遇到的問題

### 第一階段：基礎 ReAct 循環的打通
- **目標**：讓 LLM 能輸出 JSON 並由 Rust 執行 `ls`。
- **挑戰 1**：LLM 模型名稱不符。
  - **對策**：使用 `curl /v1/models` 確認正確名稱為 `openai/gpt-oss-120b`。
- **挑戰 2**：解析 JSON 失敗。
  - **對策**：改寫 `extract_json_and_execute`，使其能自動偵測內容中第一個 `{` 與最後一個 `}`。

### 第二階段：互動控制與即時反饋
- **目標**：加入 `[ESC]` 打斷功能與串流輸出。
- **挑戰 3**：`read_line` 在 Raw 模式下失效。
  - **對策**：實作**動態模式切換**。對話時 `disable_raw_mode`，任務執行時 `enable_raw_mode`。
- **挑戰 4**：終端機換行排版混亂。
  - **對策**：實作 `print_raw` 函數，將 `\n` 替換為 `\r\n`。

### 第三階段：克服 LLM 的自我限制 (防幻覺)
- **目標**：讓 Agent 順利連網並抓取 Google News。
- **挑戰 5**：嚴重的「沙箱隔離」幻覺。
  - **對策**：更新 `system.md` 強化權限提示，並實作攔截器引導其回歸 ReAct 邏輯。

### 第四階段：空間意識與能力受控化
- **目標**：解決 LLM 不聽從路徑規範的問題。
- **挑戰 6**：指令服從度遇到瓶頸。
  - **對策**：將路徑決策權收回 Rust 內核，新增 `save_script` 與 `save_plugin`。
- **挑戰 7**：對話資源監控。
  - **對策**：實作 `Usage` 解析與顯示 Token 統計。

### 第五階段：Telegram 串接、Stdin 通訊與共享上下文
- **目標**：實現手機端遠端遙控與本地端共享記憶。
- **挑戰 8**：互動式 Agent 與 Bot 通訊阻斷。
  - **對策**：新增「單次處理模式」，支援 Stdin 模擬與 Regex 解析。
- **挑戰 9**：跨介面記憶不同步。
  - **對策**：全面改為讀寫 `storage/history.md`，達成實體持久化記憶。
- **挑戰 10**：連線管理不穩定。
  - **對策**：改寫為「手動長輪詢」模式，支援自動重連。

### 第六階段：配置動態化與互動式安裝 (UX)
- **目標**：無需重新編譯即可更換 LLM 配置。
- **挑戰 11**：硬編碼配置維護成本高。
  - **對策**：使用 `std::env` 讀取 `LLM_API_URL` 並實現 Runtime Config。
- **挑戰 12**：配置過程繁瑣。
  - **對策**：升級 `manage.sh` 新增 `setup` 互動式導引。

### 第七階段：自動化進化與多步執行 (Self-Evolution)
- **目標**：讓 Agent 具備「自動造輪子」並將其「永久內化」的能力。
- **挑戰 13**：單一行動模式效率低。
  - **對策**：重構解析邏輯，支援 **JSON Array 批次執行**。
- **挑戰 14**：自我認知的更新。
  - **對策**：實作 **`promote_skill` 行動**，自動移動腳本並動態修改 `storage/system.md` 追加新技能。
- **驗證**：成功通過 `tests/test_evolution.py` 壓力測試，Agent 完成了自舉進化閉環。

## 🛠️ 重要經驗總結
1. **介面去耦與核心整合**：將大腦與介面完全切分，讓 Agent 能輕易擴展至任何平台。
2. **閉環進化 (The Feedback Loop)**：一個真正強大的 Agent 能修改自己的操作手冊。
3. **配置與代碼分離**：大幅縮短了測試循環。
4. **批次處理的魯棒性**：支援 JSON 陣列讓 Agent 能更有計畫性地執行任務。

## 🏁 結論
這個 V0 版本證明了透過簡單的 Rust 控制邏輯與強大的大模型，我們可以建立一個具備「學習能力」與「自我進化」能力的系統代理人。
