# Rust Bootstrap Agent (V0) 開發日誌

這份日誌記錄了開發過程中遇到的關鍵挑戰以及我們採取的技術對策。

## 📅 開發階段與遇到的問題

### 第一階段：基礎 ReAct 循環的打通
- **目標**：讓 LLM 能輸出 JSON 並由 Rust 執行 `ls`。
- **挑戰 1**：LLM 模型名稱不符。
  - **現象**：發送請求回傳 404，提示 `gpt-oss-120b` 不存在。
  - **對策**：使用 `curl /v1/models` 查詢實體，確認正確名稱為 `openai/gpt-oss-120b`。
- **挑戰 2**：解析 JSON 失敗。
  - **現象**：LLM 有時不加 Markdown 標籤，導致舊解析器抓不到 JSON。
  - **對策**：改寫 `extract_json_and_execute`，使其能自動偵測內容中第一個 `{` 與最後一個 `}`。

### 第二階段：互動控制與即時反饋
- **目標**：加入 `[ESC]` 打斷功能與串流輸出。
- **挑戰 3**：`read_line` 在 Raw 模式下失效。
  - **現象**：開啟 Raw 模式偵測 ESC 後，用戶輸入對話時無法按 Enter 送出。
  - **對策**：實作**動態模式切換**。對話時 `disable_raw_mode`，任務執行時 `enable_raw_mode`。
- **挑戰 4**：終端機換行排版混亂。
  - **現象**：Raw 模式下 `\n` 只下移不回首，導致文字階梯狀顯示。
  - **對策**：實作 `print_raw` 函數，手動將輸出中的 `\n` 全部替換為 `\r\n`。

### 第三階段：克服 LLM 的自我限制 (防幻覺)
- **目標**：讓 Agent 順利連網並抓取 Google News。
- **挑戰 5**：嚴重的「沙箱隔離」幻覺。
  - **現象**：LLM 雖然具備 `curl` 權限，但卻不斷聲稱「我被沙箱封鎖」，導致它拒絕執行指令。
  - **對策 1**：更新 `system.md`，使用 **ROOT 權限** 注入與強制命令模式。
  - **對策 2**：在 Rust 程式中實作 **攔截器**。當偵測到「無法回答」等關鍵字時，自動回傳提醒訊息引導其回歸 ReAct 邏輯。

### 第四階段：空間意識與能力受控化
- **目標**：解決 LLM 不聽從路徑規範的問題，並建立明確的「技能 vs 任務」區分。
- **挑戰 6**：指令服從度遇到瓶頸。
  - **現象**：LLM 雖然被告知使用 `plugins/`，但仍傾向於在根目錄或自創 `scripts/` 下使用 `cat` 命令。
  - **對策 1**：將「路徑決策權」收回 Rust 內核。新增 `save_script` 與 `save_plugin` 行動，硬性規定存檔路徑。
  - **對策 2**：更新 `system.md` 為「環境操作手冊」，賦予每個資料夾語意靈魂（核心區、記憶區、實驗室）。
- **挑戰 7**：對話資源監控。
  - **現象**：用戶無法得知本地大模型（120B）的消耗狀況與上下文剩餘空間。
  - **對策**：實作 `Usage` 解析與顯示功能，在串流結束後即時列印 Token 統計（Input/Output/Total）。

### 第五階段：Telegram 串接、Stdin 通訊與多介面共享上下文
- **目標**：實現手機端（Telegram）遠端遙控與本地端（Terminal）共享同一個對話記憶。
- **挑戰 8**：互動式 Rust Agent 與 Python Bridge 的通訊阻斷。
  - **現象**：Rust Agent 啟動後會進入 `User:` 提示符等待輸入並忽略命令列參數，導致 Telegram 端無法直接驅動 Agent。
  - **對策 1**：在 `src/main.rs` 新增「單次處理模式」，若有命令列參數則視為 User 輸入，處理完畢後直接存檔退出。
  - **對策 2**：在 `plugins/telegram_bridge.py` 實作 Stdin 模擬與 `Assistant:` 輸出解析 Regex。
- **挑戰 9**：跨介面記憶不同步。
  - **現象**：Telegram 與本地端各說各的，互不記得對方的對話內容。
  - **對策**：將記憶體中的 `Vec<Message>` 全面改為讀寫 `storage/history.md`。每當有一則訊息進來（無論來源），Agent 都會先從磁碟讀取歷史，處理完後再寫回，達成「實體持久化記憶」。
- **挑戰 10**：穩定性與連線管理。
  - **現象**：Python 的 `run_polling()` 在背景執行時容易超時或發生 `Conflict`。
  - **對策**：改寫為「手動長輪詢 (Manual Long Polling)」模式，具備自動 `offset` 偏移控制與異常自動重連機制。

### 第六階段：配置動態化與互動式安裝 (User Experience)
- **目標**：讓非開發者也能輕鬆部署，且無需重新編譯即可更換 LLM 配置。
- **挑戰 11**：硬編碼配置導致的維護成本。
  - **現象**：修改 `API_ENDPOINT` 或 `MODEL_NAME` 必須重新執行 `cargo build`，且容易誤將 Token 提交至 Git。
  - **對策 1**：在 `src/main.rs` 使用 `std::env` 讀取 `LLM_API_URL` 與 `LLM_MODEL_NAME`。
  - **對策 2**：實現 **Runtime Config**，啟動時動態加載 `.env`。
- **挑戰 12**：配置過程繁瑣不直覺。
  - **現象**：新手需手動建立多個 `.env` 檔案並查找 Token 欄位。
  - **對策**：升級 `manage.sh`，新增 **`setup` 互動式導引**。透過對話方式詢問配置並自動產生 `.env`。

## 🛠️ 重要經驗總結

7. **配置與代碼分離 (12-Factor App)**：將敏感配置與環境參數移出代碼，不僅提升了安全性（避免 Token 洩漏），更大幅縮短了測試循環——現在切換模型只需重啟程式，不需重新編譯。
8. **引導式部署 (Self-Documenting CLI)**：一個具備 `status` 與 `setup` 功能的管理腳本，能顯著降低複雜系統的進入門檻。



## 🏁 結論
這個 V0 版本證明了透過簡單的 Rust 控制邏輯配合強大的推理模型（120B），我們可以建立一個具備高度自主性與執行力的系統代理人。未來的開發方向可聚焦於更精細的工具呼叫與外掛插件系統。
